# Snap Duel — 需求文档 (V1)

## 一句话描述
两个陌生人各自发起"匹配"，系统把同时在等待的两人配成一局，进行连续多局的石头剪刀布对战，实时同步双方出拳，直到一方先到 N 胜或主动离开。

---

## 1. 核心玩法

### 1.1 匹配
- 用户打开网站，点击"开始匹配"。
- 服务器维护一个等待队列（waiting pool）。
- 如果队列里已有人在等，立刻把双方配成一局，建立房间（room），两人都进入对战页面。
- 如果队列为空，把当前用户放入队列，前端显示"正在匹配"，等待另一人加入。
- 匹配成功后，双方都不知道、也不需要知道对方的身份信息（纯匿名）。

### 1.2 对战
- 一局比赛 = 多个回合（round），先达到 **N 胜**（N 可配置，默认 3，即三局两胜或五局三胜，先定为 BO5 → 先到 3 胜）的一方获胜。
- 每个回合：
  1. 双方各自选择 石头 / 剪刀 / 布。
  2. 任一方选完后，UI 显示"已出拳，等待对方"，但不能让对方提前看到你出的是什么（直到双方都出完才同时揭示）。
  3. 双方都出拳后，服务器判定本回合胜负，推送结果给双方：本回合谁赢、当前比分。
  4. 比分未达到 N 胜时，自动开始下一回合（不需要重新匹配，不需要任何人手动点"下一局"——直接进入下一回合出拳界面）。
- 达到 N 胜后，整场比赛结束，显示最终比分和"获胜/失败"，并提供"再来一局"按钮——点击后回到匹配队列重新走一次匹配流程（不保证匹配到同一个人）。

### 1.3 离开/异常
- 任意一方主动离开（关闭页面、点击"离开"按钮）：服务器检测到后，通知另一方"对手已离开"，并把另一方送回首页。
- 断线/卡住不出拳：**V1 不做心跳检测和重连机制**，简单处理即可——如果后续要加超时判负或重连，留作 V2。
- 不需要处理"恶意挂机占用队列"之类的边界情况，V1 不考虑。

---

## 2. 不需要做的事（明确排除，避免过度设计）

- 不需要用户注册/登录/账号系统。
- 不需要持久化战绩、历史记录、排行榜。
- 不需要好友匹配、私人房间号、邀请链接对战特定的人（V1 只支持随机匹配）。
- 不需要心跳检测、断线重连、超时判负的精细逻辑。
- 不需要防作弊机制（比如检测客户端是否篡改了出拳值），V1 阶段服务器端做基本的"以服务器收到的值为准"即可，不做深度防护。
- 不需要移动端 App，只做响应式网页。

## 3. 技术方案

### 3.1 通信
- **WebSocket** 实时通信，服务器需要常驻进程维护连接状态。
- 不使用轮询、不使用 `window.storage`（那是 Claude artifact 专属 API，不适用于独立部署的网站）。

### 3.2 架构建议
- **后端**：Node.js + `ws`（或 Socket.IO，二选一，Socket.IO 自带房间和重连机制，即使 V1 不做重连，用它也方便未来扩展）。
- **状态管理**：全部状态存在服务器内存中即可（等待队列、房间、比分），不需要数据库，因为没有持久化需求。重启服务器=所有对局重置，这是可接受的。
- **前端**：纯静态页面（HTML/CSS/JS 或简单的 React），通过 WebSocket 连接后端。
- **部署**：
  - 后端需要一个支持长连接、常驻进程的托管平台（例如 Railway、Render、Fly.io；不能用纯静态托管如 GitHub Pages，也不适合用纯 serverless 函数，因为 WebSocket 需要保持连接状态）。
  - 前端可以单独部署在 Vercel/Netlify，也可以和后端一起从同一个服务里出。

### 3.3 数据结构（建议）
```
WaitingPool: [socketId, ...]   // 等待匹配的人

Room {
  roomId: string
  players: [socketIdA, socketIdB]
  scores: { [socketId]: number }
  currentRoundChoices: { [socketId]: "rock" | "paper" | "scissors" | null }
  winTarget: number   // 默认 3
}
```

### 3.4 关键事件（WebSocket 消息类型，供 Claude Code 参考设计）
- `client → server: find_match` — 请求加入匹配队列
- `server → client: match_found` — 匹配成功，附带 roomId
- `client → server: choose` — 提交本回合出拳（rock/paper/scissors）
- `server → client: round_result` — 本回合结果（双方出拳、胜负、当前比分）
- `server → client: match_result` — 整场比赛结束（最终比分、胜/负）
- `server → client: opponent_left` — 对手离开
- `client → server: leave` — 主动离开
- `client → server: rematch` — 再来一局（重新进入匹配队列）

## 5. V1 验收标准
1. 两个不同的浏览器（或两台设备）打开网站，各自点击"开始匹配"，能在几秒内被配对进入同一局。
2. 双方出拳后能实时同步揭示，胜负判定正确，比分正确累计。
3. 连续对战到一方先达到 3 胜，正确显示最终结果。
4. 任一方关闭页面或点击离开，另一方能收到提示并返回首页。
5. 点击"再来一局"能重新进入匹配流程。
